# 任务拆分（按优先级/模块）：公众号排版优化 Markdown 渲染组件

## 0. 参考文档与口径说明

本任务拆分基于以下现有文档的共同约束：

- 需求文档：requirement_analysis.md
- 架构文档：技术架构设计_公众号Markdown渲染组件.md（等价于 technical_architecture.md 的来源）
- 设计指南：设计指南_UIUX_公众号Markdown组件.md（等价于 design_guide.md 的来源）
- 原型页面：ui-design.html（用于视觉/交互对齐与验收）

目标：将项目拆成可由 AI 逐步实现的任务序列；每个模块结束后都能在 playground 中看到可验证的可视化效果或可验证的导出结果。

## 1. 优先级定义

- P0（阻塞）：项目可运行、可构建、可本地预览；决定后续迭代效率。
- P1（核心可见）：预览链路跑通（Markdown → 渲染），UI 框架成型。
- P2（体验完成）：主题/视图/代码块样式达到设计指南要求。
- P3（关键业务）：复制为公众号格式（内联样式 + 白名单）并可验证兼容性。
- P4（可集成）：导出控制按钮能力、API 稳定、对外集成体验完善。
- P5（交付）：发布与质量保证（lint/typecheck/build/test、版本策略、文档收敛）。

## 2. 总体模块拆分（从可见到可交付）

### 模块 A（P0）项目脚手架与工作区（npm + Vite + React）

**目标**：统一 npm 管理，建立 workspaces，跑起 playground；为组件库构建打底。

**任务 A1：仓库初始化（workspaces 骨架）**
- 产出
  - 根 package.json（npm workspaces：packages/*、apps/*）
  - 根级脚本：dev / build / build:lib / build:play / lint / typecheck
  - 统一锁文件：package-lock.json
- 验收/可见效果
  - `npm install` 可成功
  - `npm run dev` 能启动 playground 并打开页面

**任务 A2：playground 工程创建（Vite + React）**
- 产出
  - apps/playground：最小可运行页面
  - 页面包含：顶部区域占位 + 预览容器占位 + Markdown 输入区占位
- 验收/可见效果
  - 浏览器可访问本地地址，看到基础布局（无需主题与渲染）

**任务 A3：组件库工程创建（Vite library mode）**
- 产出
  - packages/md-wx-renderer：库模式构建产物（ESM/CJS + d.ts）
  - React/ReactDOM peerDependencies 配置
- 验收/可见效果
  - `npm run build:lib` 输出构建产物
  - playground 可从库入口导入一个占位组件并渲染（验证 workspace 链路）

### 模块 B（P1）Markdown 渲染最小闭环（实时预览）

**目标**：满足 requirement_analysis.md 的“受控输入、实时渲染”，先把预览链路跑通。

**任务 B1：Preview 组件最小实现（受控 markdown 输入）**
- 产出
  - `<Preview markdown="..." />`：接收 markdown 字符串，实时更新预览
  - playground：左侧 textarea 输入，右侧实时预览输出（最小集：标题/段落/列表/引用/代码块）
- 验收/可见效果
  - 输入变化后 300ms 内可见更新（肉眼验证）

**任务 B2：Markdown 能力与安全策略（默认安全）**
- 产出
  - 明确的“是否允许原生 HTML”开关（默认关闭）

- 验收/可见效果
  - 输入包含 `<script>` 或事件属性时，预览不执行脚本，输出被过滤/转义

**任务 B3：基础排版（非主题版）**
- 产出
  - 预览区基础排版（字体、行高、段落间距、标题层级）
- 验收/可见效果
  - 同一份 Markdown 在预览区可读性良好，结构层级清晰

### 模块 D（P2）主题系统（5 主题 + token）与内容风格联动

**目标**：落实 requirement_analysis.md 与 design_guide.md：5 个主题，切换后标题/引用/列表/强调/链接等整体风格变化且平滑过渡。

**任务 D1：主题模型与 token 规范**
- 产出

    - t-juejin：掘金酱紫
    - t-wechat-dark：樱花粉（命名保持兼容，但展示名为樱花粉）
    - t-tech-blue：科技蓝
    - t-vintage-orange：复古橙
  - token 集合（--t-*）覆盖：surface/text/heading/quote/list/link/inline-code/codeblock/typography
- 验收/可见效果
  - 切换主题时预览区颜色与字体明显变化

**任务 D2：主题下拉菜单（🌈）与状态展示**
- 产出
  - 下拉项显示主题名 + 色块；当前主题有选中态
  - 主题切换触发预览区整体样式变化（含平滑过渡）
- 验收/可见效果
  - 与 ui-design.html 的交互一致（展开位置、动画、选中态）

**任务 D3：Markdown 元素的主题化样式覆盖**
- 产出
  - H1–H6：字号/字体/颜色随主题变化
  - blockquote：背景、左边线、文字颜色随主题变化
  - list marker：随主题变化
  - strong/em：颜色/风格随主题变化
  - a：下划线/颜色/hover 颜色随主题变化
  - inline code：背景/边框/文字随主题变化
- 验收/可见效果
  - 使用同一份示例 Markdown，切换主题可明显看到上述元素均被影响

### 模块 H（P3）复制为微信公众号格式（最关键）

**目标**：实现“复制预览组件内容为公众号支持的 HTML（内联样式）”，并通过白名单与降级策略确保兼容性。

**任务 H1：导出 HTML 结构定义（Export 层）**
- 产出
  - `exportToWechatHtml(markdown, theme, viewport, options?)`：输出 HTML 字符串
  - 输出只包含白名单标签（design_guide.md 第 7 节）
- 验收/可见效果
  - playground 增加“导出预览”面板：显示导出的 HTML（便于人工检查）

**任务 H2：内联样式生成（主题 token → inline style）**
- 产出
  - 将主题 token 映射为每类元素的内联 style（p/h1..h6/blockquote/li/a/inline-code/pre/code 等）
  - 不依赖 class/id；不使用高风险属性（position/flex/transform 等）
- 验收/可见效果
  - 切换主题后，导出 HTML 的 style 串随之变化，且结构一致

**任务 H3：Sanitizer（白名单过滤 + 归一化）**
- 产出
  - HTML 标签/属性/style 白名单
  - 移除 class/id/on* 事件与不支持属性；style 归一化（单位/颜色/重复属性）
- 验收/可见效果
  - 在导出面板中可见不合法标签/属性被清理

**任务 H4：剪贴板写入（text/html + text/plain 兜底）**
- 产出
  - 点击 📋 后写入剪贴板 HTML
  - 不支持时降级为复制纯文本或复制 HTML 字符串
- 验收/可见效果
  - 点击复制后出现可感知反馈（toast）
  - 将结果粘贴到富文本编辑器（或微信公众平台编辑器）后，基本排版可用

**任务 H5：链接与图片策略（保守默认）**
- 产出
  - 链接默认保守策略：可配置仅允许 mp.weixin.qq.com 或降级为样式化文本
  - 图片仅 https，保留 alt，限制最大宽度避免破版
- 验收/可见效果
  - 导出 HTML 的 a/img 符合策略；不出现不受控外链风险

**任务 H6：代码块复制降级策略**
- 产出
  - 复制输出中代码内容保持纯文本；装饰结构允许降级移除
  - 高亮 token 被过滤时仍能保持等宽、背景、行高等可读性
- 验收/可见效果
  - 粘贴后的代码块仍可读，不出现内容错位或“行号 DOM 污染”
